# 2025년 9월 1일

## 🚩 문제 상황

• 문서화 미흡
    ◦ 테이블 정의서는 일부 작성되었지만, 컬럼 정의는 벤더사 직접 문의가 필요할 정도로 내부 자료가 없음.
    ◦ Colocation 환경은 문서화가 전혀 안 되어 있어 사실상 무인도처럼 방치.
• 중복·비효율 운영
    ◦ Koscom 데이터는 여러 팀이 동일 데이터를 별도 서버에 적재, 파일 관리 부재로 쓰레기 데이터가 쌓임.
    ◦ NAS는 리서처들이 데이터를 쉽게 받도록 공유되지만, 파일 생성/관리 기준이 없어 무질서하게 방치됨.
• 조직/팀 단절
    ◦ 팀 간뿐 아니라 팀 내부에서도 데이터 사용·저장 방식이 완전히 분리 → 서로 어떤 데이터를 쓰는지 파악 불가.
    ◦ 인수인계 부재, 조직 개편 반복으로 팀명칭과 책임 소재가 불분명. (예: A팀 산하 b, c, d 중 누가 관리하는지 알 수 없음, 퇴사로 단절된 경우 다수)
• 의사소통/협력 문제
    ◦ 일부 구성원은 요청사항에 거부감을 보이며 "자기 방식대로 잘 되고 있다"고 인식 → 표준화 무시.
    ◦ 실제로는 서비스로 나가는 테이블임에도 데이터 출처 확인조차 안 되는 사례 발생.
    ◦ AI Product팀의 경우, Refinitiv 데이터 종료를 결정했으나 vw_security 테이블에서 사용 중이어서 혼란 발생.

## 🔧 대처 방법

• 테이블 정의서를 작성해 관리의 첫걸음을 마련.
• 팀별로 데이터 출처와 사용 여부를 확인하기 위해 적극 소통.
• 벤더사와 직접 커뮤니케이션해 컬럼 정의 등 미비한 메타데이터를 확보.

## ✨ 성과 / 개선점

• 성과: 데이터 출처 확인과 문서화 작업을 시작하여 파편화된 상황에 대응.
• 성과: 팀별 엔지니어와의 소통을 통해 일부 데이터 사용 현황을 파악.
• 개선: 전사 차원의 문서화·거버넌스 체계가 전무하여, 여전히 데이터 관리 리스크가 큼.

## 🤝 배운 점

• 데이터 관리가 팀별 사일로에 갇히면, 서비스 품질뿐 아니라 의사결정 자체가 혼선에 빠진다.
• 결국 문서화와 출처 관리가 데이터 거버넌스의 핵심이며, 이를 소홀히 하면 누구도 책임질 수 없는 상태에 빠진다.

## 💡 한 줄 요약

• 팀별 단절과 문서화 부재로 인한 데이터 혼선을 점검하고, 출처 중심의 데이터 거버넌스 정립 필요성을 확인했다.

## 추가 내용

- 테이블 정의서 작성
- 팀별 소통, 데이터가 출처 확인 소통
- 컬럼 정의서는 벤더사에 직접 문의해야함
- Koscom 데이터는 여러팀에서 같은 데이터임에도 불구하고 별도의 서버를 띄워서 적재하고 있음. 그리고 그 서버 파일가 안되서 쓰레기통이 따로 없음.
- colocation 경우, 문서화가 제대로 안되어 있어서 사실상 무인도와 같이 운영되고 있음. 인수인계도 제대로 안되서 전임자가 데이터를 받아서 처리하는 부분을 관리하지도 않고 심지어 그 데이터를 계속 받아서 쓰는 곳도 있음.
- 모든 데이터는 리서처들이 다운로드받기 쉽게 nas에다가 옮겨 놓는데 이것조차 파일관리가 제대로 안되어 있어서 마음대로 생성하고 방치함.
- 팀간 뿐만 아니라 팀 내부에서도 각자 완벽히 분리되어 있어서 어떤 데이터를 쓰고 어떻게 저장되고 있는지 전혀 모르고 있음.
- 그나마 각 팀에서 데이터 엔지니어를 겸하는 분이 계시면 그 분과 소통해서 원활하게 파악할 수 있는 반면, 아닌 곳은 팀이지만 사실상 각자도생임.
- 조직개편도 일부 사업부분에서 반복적으로 일어나서 사용하는 팀명칭도 서로 너무 다름. A - b, c, d. A가 한거라서 모른다. b, c, d팀에게 전부다 물어봐야 함. 그 중에서도 퇴사해서 모르는 경우도 있음
colocation 해외선물 데이터 처리부분
- 요청사항에 대해서 상당한 거부감이 있는 사람도 있음. 본인은 자기 방식대로 잘하고 있어서 신경을 안써도 된다. 자기팀은, 사실 팀끼리도 소통이 안되지만, 잘되고 있다.
- 서비스로 나가는 테이블인데도 데이터 출처도 모르고 쓰는 경우도 있었음
AI Product팀 vw_security 테이블, refinitiv 데이터 종료하려고 하는데 알고보니 사용하고 있어서 의사결정 혼동옴
