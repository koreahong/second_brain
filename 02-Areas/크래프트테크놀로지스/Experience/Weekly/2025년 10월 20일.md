# 2025년 10월 20일

## 🚩 문제 상황

• Airflow DAG가 산발적으로 관리되어 유지보수성이 낮고, Variable/Alert 로직이 중복되어 구조적 개선 필요
• 코드 컨벤션 및 자동 변환 기준이 부재해 Claude Code 등 AI 도구를 활용하기 어려움
• FactSet 기반 대체 데이터 도입 논의 중, 리서처 요구사항과 벤더 제공 포맷 간 간극 존재
• 팀장 이직 및 신규 팀원 합류로 인한 업무 인수인계 및 기술 기준 재정비 필요

## 🔧 대처 방법

1. Claude Code 도입 및 코드 관리 연구
    ◦ Airflow DAG / Python 모듈 자동 변환 시 컨벤션 및 포맷 통일 방안 검토
    ◦ 코드 리뷰 자동화 및 리팩토링 기준 수립 실험 (docstring, typing, 구조 일관성 등)
    ◦ 기존 utils 기반 얼럿 및 커스텀 함수 구조 분석 후 단계별 제거 계획 수립
2. DAG 레거시 청산 프로세스
    1. 기존 DAG 신규 구조로 마이그레이션
    2. Variable → 하드코딩 전환 (환경 의존도 제거)
    3. utils 기반 얼럿 및 불필요한 헬퍼 삭제
    4. SQL 실행 로직을 파일 단위(.sql)로 분리 후 커스텀 커넥터를 통한 실행 구조로 전환
    5. 이후 빌드/변환 단계는 DBT로 이관 예정
3. 대체 데이터(FactSet) 검토
    ◦ 리서처 요구사항 정리 및 제공 데이터 스펙 매칭
    ◦ FactSet 담당자와 미팅을 통해 파일 스펙, 제공 주기, API 연동 방식 파악
    ◦ 기존 LSEG/Refinitiv 데이터와 비교 표 작성
4. 조직 변화 대응
    ◦ 팀장 이직에 따른 업무 분배 재정비
    ◦ 신규 팀원 온보딩 문서 및 코드 가이드라인 업데이트

## ✨ 성과 / 개선점

성과:
• DAG 정리 및 구조 통일로 관리 효율성 향상
• 코드 표준화 초석 마련으로 AI 기반 개발 자동화의 기초 확보
• FactSet 데이터 검토를 통해 대체 데이터 전략 방향 명확화
• 팀 체계 재정비로 인수인계 리스크 최소화
개선:
• Claude Code 기반 코드 자동 변환 시 컨텍스트 손실(주석, naming convention) 개선 필요
• DAG 변환 과정에서 일부 파이프라인 의존성 문제 발생

## 🤝 배운 점

• “자동화의 효율성은 표준화 위에서만 가능하다”는 점을 실감
• 코드 변환보다 중요한 것은 컨벤션의 정의와 일관된 관리 체계
• 데이터 공급자와 내부 사용자 간 요구 명세의 세밀한 조율이 필수

## 💡 한 줄 요약

AI 기반 코드 자동화와 레거시 정리의 교차점에서, 표준화된 개발 체계 확립을 위한 전환기.
