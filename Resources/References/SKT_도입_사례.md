# SKT 도입 사례

## 📝 내용

### 도입 이유 (Why OS = Object Storage 서비스 필요했는가)

1. 스크래치에서 대용량 비정형 데이터 저장

1. 부서 간 및 사용자 간 데이터 공유 비효율성 개선

1. 보안 / 비용 절감 요구

1. 운영 및 확장성 / 가용성 확보

1. API / 연동성

### MinIO + Kubernetes 구성을 선택한 이유 및 장점

## 우리 회사와의 유사점 & 시사점

- 여러 부서에서 각기 다른 저장소 사용 중이고, 데이터 사일로(silo)가 있음 → 데이터 공유/협업/중복 저장 등의 비효율 존재

- “디렉토리 + 파일” 수준 메타데이터만 있고, 테이블·버킷 수준 관리·라인리지(traceability)가 부족함

- 데이터 거버넌스/메타데이터 관리 툴 (예: DataHub) 도입 또는 검토 중임

- 저장 데이터의 규모/종류가 늘어나고 있고, 대용량 · 비정형 데이터 (로그, 이미지, 동영상 등) 증대 추세임

이런 점에서 SK Telecom의 사례는 우리에게 매우 참고 가능함.

## 핵심 도입 이유 + 우리 회사에서 예상 가능한 효과

- 메타데이터 + 구조화된 데이터 관리 가능 → DataHub 같은 툴이 디렉토리+오브젝트 단위 메타데이터 수집 및 관리 용이

- 데이터 사일로 탈피: 중앙저장소(OS)로 데이터 중앙화, 부서 간 공유 개선

- 보안/권한 관리 강화: 외부 저장/물리적 매체 사용 줄이고 내부 인증/인가 통해 접근 통제

- 비용 절감: 외부 클라우드 스토리지 / 반복 저장 / USB 등 물리장치 관리비용 절감

- 운영 안정성 확보: 장애 대비(디스크, 노드), 모니터링/알람, 확장 가능성
